# Bid On Flow Console

비공개 원본 프로젝트의 실시간 경매 처리 순서를 공개 가능한 가상 데이터로 재구성한 정적 기술 시연 도구입니다. 실제 Flutter 앱 UI나 운영 모니터링 화면이 아니며 실제 서버, 사용자, 로그와 연결되지 않습니다.

## 포함 시나리오

- 정상 입찰
- 두 사용자의 동시 입찰
- 마감 직전 자동 연장
- WebSocket 이벤트 누락과 상태 복구
- 경매 종료 중복 요청 차단
- 거래 생성 실패 후 재처리 흐름

각 시나리오는 공개 문서에서 확인한 처리 순서와 검증 상태를 보여줍니다. 금액·시간·Version·사용자는 설명용 가상 값이며 처리 시간은 성능 측정값이 아닙니다.

## 실행 방법

Node.js 20 이상을 기준으로 합니다.

```bash
npm ci
npm run dev
```

개발 서버가 안내한 로컬 주소에서 실행합니다.

## 테스트와 빌드

```bash
npm run type-check
npm run test
npm run build
```

`dist/`에 정적 파일이 생성됩니다. GitHub Pages Workflow는 저장소명을 이용해 `VITE_BASE_PATH`를 주입하므로 로컬은 `/`, Pages는 `/Bid/` 같은 하위 경로에서 동작합니다.

## Query Parameter 딥링크

```text
?scenario=concurrent-bid
?scenario=websocket-recovery
?scenario=duplicate-finish
```

유효하지 않은 값은 정상 입찰로 돌아갑니다. 시나리오 버튼을 누르면 `history.pushState`로 URL을 갱신하며 새로고침과 브라우저 앞·뒤 이동에서도 선택을 유지합니다. 별도 서버 라우팅은 사용하지 않습니다.

## 시나리오와 공개 문서

시나리오의 `관련 기술 문서 열기`는 GitHub의 공개 문서로 이동합니다. 정적 Pages에서 Markdown 상대 경로가 앱 경로로 해석되는 문제를 피하기 위해 저장소 URL을 사용합니다.

## 시나리오 추가 방법

1. `src/query.ts`의 허용 ID에 값을 추가합니다.
2. `src/scenarios.ts`에 가상 상태, 단계, 근거와 문서 링크를 추가합니다.
3. 딥링크 테스트를 추가하고 type-check, test, build를 실행합니다.
4. 원본 프로젝트에서 확인되지 않은 동작이나 검증 결과는 추가하지 않습니다.

## 공개 데이터 기준과 현재 한계

- 실제 API 주소, Redis Key, 메시지 채널, 인증 정보, 사용자 데이터와 운영 로그를 포함하지 않습니다.
- 실제 Flutter 화면과 무관한 기술 시연 UI입니다.
- 네트워크 요청 없이 브라우저 안에서 정적으로 재생합니다.
- `코드에서 구현 확인`, `테스트 코드 확인`, `흐름 재구성`, `추가 검증 필요`를 구분합니다.
- 실제 장애 환경, 네트워크, 동시 부하와 다중 인스턴스 통합 검증을 대신하지 않습니다.

대표 이미지 촬영은 1440×900에서 `?scenario=concurrent-bid`를 열고 흐름을 재생한 뒤 브라우저 경로나 개발자 도구가 보이지 않게 진행합니다.
