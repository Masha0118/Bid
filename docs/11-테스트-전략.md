# 테스트 전략

테스트 파일이 존재하는 것과 특정 배포 환경에서 실행에 성공한 것은 구분합니다.

| 영역 | 확인한 테스트 | 현재 상태 |
|---|---|---|
| 직접 입찰 UI | 입력 금액 전달과 확인 정보 없음 표시 | 자동화 테스트 존재 |
| 성능 설정 UI | 제한값과 설정 입력 | 자동화 테스트 존재 |
| 경매 체험 세션 | 실제 입찰 중계, 참가자 익명화, 종료 정리 | 자동화 테스트 존재 |
| 실행 제한 | 동시 실행 잠금과 제한값 검증 | 자동화 테스트 존재 |
| 성능 결과 | k6 결과 파싱과 실행 상태 관리 | 자동화 테스트 존재 |
| 정합성 | 최종 상태 비교 | 자동화 테스트 존재 |
| 입찰 규칙 | 판매자·최고 입찰자·금액·상태 조건 | 자동화 테스트 존재 |
| Redis 상태 | 입찰 저장과 상태 복원 | 자동화 테스트 존재 |
| WebSocket | 실시간 묶음 처리와 Version 흐름 | 자동화 테스트 존재 |
| Kafka | 발행, 소비, 중복 처리와 통합 경로 | 자동화 테스트 존재 |
| RabbitMQ | 예약 메시지 구성과 수신 처리 | 자동화 테스트 존재 |
| 거래 중복 방지 | 같은 낙찰 결과 재처리 | 자동화 테스트 존재 |
| PostgreSQL | 저장소와 제약의 통합 동작 | Testcontainers 기반 테스트 존재 |

## 격리 환경에서 확인된 범위

직접 입찰, 가상 참가자 입찰, 네 참가자의 WebSocket 상태 동기화와 처리 단계 수집은 격리 Docker 환경의 통합 검증 기록에서 확인했습니다. 현재 공개 배포 화면은 접속되지만 실행 기능이 비활성 상태이므로 이번 작업에서 동일한 통합 실행을 재현했다고 쓰지 않습니다.

## k6

k6 시나리오는 한 경매에 여러 사용자가 입찰하는 경쟁 상황을 생성합니다. 결과는 전체 요청, 반영된 입찰, 조건상 미반영, 오류, 지연 시간과 처리량으로 나눠 집계합니다. 테스트 환경 사양과 실행 설정이 없는 수치는 비교 근거로 사용하지 않습니다.

## 추가 검증 필요

- 장시간 WebSocket 연결과 강제 재연결
- Redis·Kafka·RabbitMQ 재시작을 포함한 장애 주입
- 다중 Auction Server 환경의 이벤트 순서와 상태 수렴
- 대규모 다중 경매 부하
- Kafka 전달 이후 최종 저장 완료의 실행별 집계
- 경매 종료부터 거래 생성까지의 전체 통합 재처리

## 권장 실행 순서

1. 프론트엔드 단위 테스트와 타입 검사
2. 경매 실행 제어 서비스 단위 테스트
3. Auction Server 입찰·Redis·메시징 테스트
4. PostgreSQL 통합 테스트
5. 격리 Docker 환경의 직접·가상 입찰 통합 검증
6. k6 성능 실행과 최종 정합성 확인
