# 메시징 구조

Kafka와 RabbitMQ는 모두 비동기 메시징에 사용되지만 해결하는 문제가 다릅니다.

## Kafka: 입찰 이력 전달

- 이미 판정된 입찰 결과를 비동기 후속 경로로 전달합니다.
- 사용자 응답과 이력 저장을 분리합니다.
- 성공과 조건상 미반영 결과를 감사 가능한 이벤트로 남길 수 있습니다.
- 중복 소비를 고려해 이력 저장 경로는 같은 이벤트를 여러 번 처리해도 결과가 늘어나지 않도록 설계합니다.

기본 체험의 처리 과정에서는 Kafka 전달 요청 단계까지만 확인합니다. 실행별 최종 저장 완료·대기·실패 수는 현재 확인 정보가 없으므로 저장 완료로 확대해 표현하지 않습니다.

## RabbitMQ: 예약된 시작과 종료

- 정해진 시점에 실행해야 하는 경매 시작·종료 작업을 전달합니다.
- 예약 시간과 실제 실행 로직을 분리합니다.
- 재전달 가능성을 고려해 종료 처리는 잠금과 기존 완료 결과 확인을 함께 수행합니다.

RabbitMQ 예약 흐름은 기본 웹 체험에 포함되지 않습니다. 종료 처리 문서와 자동화 테스트 근거로 구분합니다.

## 역할을 구분한 이유

| 요구 | 선택 | 이유 |
|---|---|---|
| 특정 시점의 작업 실행 | RabbitMQ | 예약 작업 전달과 재시도 흐름에 집중 |
| 발생한 입찰 결과의 연속 전달 | Kafka | 이벤트 스트림과 비동기 소비에 집중 |

입찰 성공 응답은 메시지 저장 완료를 기다리지 않습니다. 이 분리는 응답 지연을 줄이지만 메시징 장애 때 재시도·중복 방지·관측이 필요하다는 의미이기도 합니다.

## 실패 처리 방향

- 발행 실패: 재시도하고, 계속 실패하면 별도 실패 기록 경로를 사용합니다.
- 중복 전달: 이벤트 식별자를 기준으로 중복 저장을 방지합니다.
- 소비 지연: 사용자 입찰 판정과 이력 반영 시간을 분리해 관측합니다.
- 예약 작업 재전달: 종료 잠금과 완료 결과 확인으로 중복 영향을 줄입니다.

내부 Topic, Queue와 Consumer Group 이름은 문서에 기록하지 않습니다.
