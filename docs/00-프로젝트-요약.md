# Bid On 프로젝트 요약

Bid On은 동시에 들어오는 입찰 요청을 안전하게 처리하고 변경된 경매 상태를 여러 참가자에게 실시간으로 전달하는 인터랙티브 경매 시스템입니다.

[실시간 경매 직접 체험](https://bid-on-performance-lab.pages.dev/) · [빠른 시작](01-빠른-시작.md) · [시스템 아키텍처](02-시스템-아키텍처.md)

## 해결하려는 문제

실시간 경매에서는 여러 요청이 같은 최고가를 기준으로 동시에 판정될 수 있습니다. 서버의 최종 상태가 정확해도 각 참가자 화면이 다른 이벤트 순서를 받으면 서로 다른 값을 표시할 수 있습니다. 예약된 종료 작업은 중복 전달이나 후속 처리 실패까지 고려해야 합니다.

Bid On은 이 문제를 세 경계로 나눕니다.

1. 입찰 조건 확인과 상태 변경은 Redis Lua에서 한 번에 처리합니다.
2. 변경 결과는 WebSocket으로 전달하고 Version으로 최신 여부를 판단합니다.
3. 예약 작업과 입찰 이력은 RabbitMQ와 Kafka의 서로 다른 비동기 흐름으로 분리합니다.

## 직접 체험 가능한 기능

- 새 테스트 시작과 종료
- 금액 입력 후 직접 입찰
- 2~10명의 가상 참가자 실행
- 느림·보통·빠름 입찰 속도
- 순차·경쟁·유효 범위 분산 입찰 패턴
- 현재가, 최고 입찰자, 입찰 횟수와 Version 확인
- 네 참가자 카드의 WebSocket 상태 확인
- Redis Lua, WebSocket과 Kafka 전달 요청 단계 확인
- 평균·p95 응답 시간과 현재 처리량 확인

## 핵심 기술 세 가지

### Redis Lua

입찰 가능 여부와 최고가·최고 입찰자·입찰 횟수·Version 변경을 한 실행 안에서 처리합니다. 원자 범위는 Redis 내부 상태 변경까지이며 후속 메시징과 영속 저장은 별도 흐름입니다.

### WebSocket + Version

상태 전체를 매번 조회하지 않고 변경 결과를 구독 중인 참가자에게 전달합니다. Version은 중복·역순·누락 가능성을 판정하는 기준입니다.

### 역할별 메시징

RabbitMQ는 정해진 시점의 시작·종료 작업을 전달하고, Kafka는 이미 판정된 입찰 결과의 이력 전달을 담당합니다.

## 대표 처리 흐름

```text
직접 입찰·가상 참가자 → 입찰 요청 처리 → Redis Lua 판정
                                      ├→ WebSocket 상태 전달
                                      └→ Kafka 입찰 이력 전달
```

## 확인 가능한 결과

- 요청이 반영됐는지 또는 경매 조건 때문에 반영되지 않았는지
- 최종 현재가와 최고 입찰자
- 입찰 횟수와 상태 Version
- 각 참가자 카드의 Version 일치 여부
- 시도·완료 요청 수, 평균·p95 응답 시간과 현재 처리량
- 처리 단계별 관측 건수

Kafka 최종 저장 완료 수와 WebSocket 재연결 자동 복구는 기본 체험에서 확인하지 않습니다. 확인할 수 없는 수치를 0으로 표시하지 않고 `확인 정보 없음`으로 구분합니다.

## 현재 범위

현재 웹 체험은 한 경매의 직접·가상 입찰과 참가자 상태 동기화에 집중합니다. 대규모 장시간 부하, 장애 주입, 다중 서버 확장과 예약 종료 이후 전체 후속 처리는 별도 검증 범위입니다.
