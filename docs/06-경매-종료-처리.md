# 경매 종료 처리

[경매 종료 다이어그램](../diagrams/auction-finish.mmd)

## 예약된 종료 시점

Main API는 경매 시작과 종료처럼 특정 시점에 실행해야 하는 작업을 RabbitMQ에 전달합니다. Auction Server는 예약 시간이 된 작업을 받아 실제 경매 상태를 확인한 뒤 종료 처리를 시작합니다.

## 종료 처리 흐름

1. 종료 작업을 수신합니다.
2. 경매별 분산 잠금을 시도합니다.
3. 이미 완료된 종료 결과가 있는지 확인합니다.
4. Redis의 최종 런타임 상태를 조회합니다.
5. 필요한 정보가 없으면 PostgreSQL의 영속 상태를 보완 조회합니다.
6. 낙찰 또는 유찰 결과를 결정합니다.
7. Main API에 후속 결과를 반영합니다.
8. 후속 반영 성공을 확인한 뒤 런타임 상태를 정리합니다.

## 중복 종료 방지

메시지가 재전달되거나 여러 실행 주체가 동시에 종료를 시도할 수 있습니다. 경매별 잠금으로 동시 실행을 줄이고, 잠금을 얻은 뒤에도 기존 완료 결과를 다시 확인합니다. 같은 낙찰 결과의 후속 요청은 기존 거래 결과로 수렴하도록 처리합니다.

## 실패와 재시도

- 잠금을 얻지 못하면 다른 종료 처리가 진행 중인 것으로 보고 중복 실행을 피합니다.
- Main API 반영이 실패하면 성공 전에 Redis 런타임 상태를 지우지 않습니다.
- 이미 생성된 거래가 있으면 새 거래를 중복 생성하지 않고 기존 결과를 사용합니다.
- 영속 상태와 런타임 상태가 다르면 보완 조회와 재처리 근거를 남깁니다.

이 흐름이 모든 장애에서 자동 복구를 보장한다는 의미는 아닙니다. 브로커 재시작, 장시간 네트워크 단절과 다중 서버 동시 실패는 별도 통합 검증이 필요합니다.

## 기본 체험 포함 여부

기본 웹 체험의 `테스트 종료`는 격리된 테스트 데이터 정리를 수행합니다. RabbitMQ 예약 종료, 낙찰 거래 생성과 전체 재처리 흐름은 기본 체험에 포함되지 않으며 코드·자동화 테스트와 상세 다이어그램에서 확인합니다.
