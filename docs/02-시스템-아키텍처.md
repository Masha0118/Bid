## 핵심 흐름

1. 참가자가 Bid On Web에서 경매를 시작하거나 입찰합니다.
2. 경매 실행 제어가 필요한 테스트 데이터를 준비하고 Auction Server에 요청합니다.
3. Auction Server가 Redis Lua에서 입찰 조건과 상태 변경을 한 번에 처리합니다.
4. 반영 결과를 WebSocket으로 참가자에게 전달합니다.
5. 입찰 이력은 Kafka를 통해 비동기 후속 경로로 전달합니다.
6. 예약된 시작과 종료는 RabbitMQ가 Auction Server로 전달합니다.
7. 장기 보관이 필요한 상품·거래·결과는 PostgreSQL에 저장합니다.

## 책임 경계

Redis Lua가 보장하는 원자 범위는 Redis 내부의 입찰 판정과 상태 변경까지입니다. WebSocket 전달, Kafka 이력 전달과 PostgreSQL 반영은 후속 처리이므로 하나의 분산 트랜잭션으로 표현하지 않습니다.

웹 체험은 격리된 실행 환경을 사용합니다. 브라우저가 내부 서버 주소나 인증 값을 선택하지 못하도록 입력 계약을 제한하고, 실행 종료 후 테스트 데이터를 정리합니다.
