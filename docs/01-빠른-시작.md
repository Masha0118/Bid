# 빠른 시작

[Bid On 웹 화면 열기](https://bidon-performance.duckdns.org/)

> 화면에 `실행 비활성화`가 표시되거나 서버 상태 확인이 끝나지 않으면 체험 백엔드가 연결되지 않은 상태입니다. 이 경우 기능을 실행했다고 판단하지 말고 배포 상태를 먼저 확인합니다.

## 1. 경매 준비

`준비할 가상 입찰자`를 2~10명 범위에서 선택하고 `새 테스트 시작`을 누릅니다. 준비가 끝나면 간이 경매의 현재가, 최소 다음 입찰가, 입찰 횟수와 Version이 표시됩니다.

## 2. 직접 입찰

`입찰가`에 최소 다음 입찰가 이상의 금액을 입력하고 `직접 입찰하기`를 누릅니다. `+1천`, `+5천` 버튼으로 금액을 올릴 수도 있습니다.

확인할 내용:

- 입찰 결과 메시지
- 현재가와 최고 입찰자 변경
- 입찰 횟수와 Version 증가
- 체험 결과의 시도·반영 수 증가

## 3. 가상 참가자 설정

`가상 입찰자 제어`에서 참가자 수, 입찰 속도와 패턴을 선택합니다.

- 느림: 2초 간격
- 보통: 1초 간격
- 빠름: 0.4초 간격
- 순차: 금액을 차례로 높임
- 경쟁: 같은 시점의 최소가로 경쟁
- 유효 범위 분산: 유효한 범위에서 서로 다른 금액 사용

## 4. 가상 입찰 시작

`가상 입찰 시작`을 누릅니다. 여러 요청이 비슷한 시점에 도착하면서 일부는 반영되고 일부는 현재가나 최고 입찰자 조건 때문에 반영되지 않을 수 있습니다. 중지하려면 `가상 입찰 중지`를 누릅니다.

## 5. 참가자 상태 확인

`참가자 화면 동기화`의 네 카드를 비교합니다. 각 카드가 `구독 중`이고 현재가, 최고 입찰자와 Version이 같으면 같은 최신 상태를 표시하고 있는 것입니다.

## 6. 처리 과정 확인

`실제 데이터 흐름`에서 다음 단계를 확인합니다.

1. 직접 입찰과 가상 참가자의 요청 생성
2. 경매 요청 처리
3. Redis Lua의 입찰 규칙 확인과 상태 변경
4. WebSocket 참가자 상태 전달
5. Kafka 입찰 이력 전달 요청

수집 계약이 없는 값은 `확인 정보 없음`으로 표시됩니다.

## 7. 실행 결과 확인

`체험 결과 요약`에서 시도한 입찰, 반영된 입찰, 조건상 미반영, 시스템 오류, 평균·p95 응답 시간과 현재 초당 처리량을 확인합니다. 참가자 카드의 Version 일치 여부도 문장으로 표시됩니다.

더 상세한 k6 설정과 정합성 검사는 `고급 성능 보기`에서 확인합니다.

## 8. 종료 또는 초기화

진행 중에는 `테스트 종료`를 눌러 테스트 데이터를 정리합니다. 종료가 끝난 뒤 `초기화`를 누르면 새 경매를 준비할 수 있습니다. 종료 직후에는 안전 대기 시간이 적용될 수 있습니다.
