# 메시징과 데이터 책임

## Redis

- 문제: 여러 입찰 요청이 같은 경매 상태를 동시에 변경
- 데이터: 최고가, 최고 입찰자, 종료 시각, 버전, 입찰 기록 등 런타임 상태
- 방식: 동기 Lua 실행과 Pub/Sub
- 실패 처리: 입찰 거절 코드 반환, 영속 백업에서 상태 복원 경로
- 한계: PostgreSQL과 단일 트랜잭션이 아니며 장애 복구 통합 시험 필요

## WebSocket

- 문제: 바뀐 경매 정보를 화면에 즉시 전달
- 데이터: 버전과 변경된 가격·입찰·종료 상태
- 방식: 비동기 서버 전파와 클라이언트 수신
- 실패 처리: 버전 누락 감지 시 REST 전체 조회
- 한계: 실제 불안정 네트워크와 장시간 재연결 시험 필요

## RabbitMQ

- 문제: 미래 시각의 경매 시작·종료 작업 전달
- 데이터: 경매 식별과 작업 시점에 필요한 최소 메시지
- 방식: 비동기 예약 작업
- 실패 처리: 리스너 예외 전파, 재시도·실패 보관 구성이 존재
- 한계: 브로커 장애와 예약 메시지 누락은 주기적 종료 재탐색으로 보완하지만 전체 재현 시험 필요

## Kafka

- 문제: 입찰 성공·실패를 사용자 응답 경로와 분리해 감사 기록
- 데이터: 이벤트 식별자, 입찰 결과와 처리 시각
- 방식: 비동기 발행·소비
- 실패 처리: 브로커 ACK 확인, 제한된 재시도, 로컬 실패 보관과 시작 시 재전송, 실패 이벤트 분리 처리
- 한계: 로컬 파일 기반 실패 보관은 다중 인스턴스 공유 저장소가 아니며 운영 보존 정책을 별도로 정해야 함

## PostgreSQL

- 문제: 상품, 사용자, 거래, 종료 결과, 감사와 관리 데이터를 장기 보관
- 데이터: 영속 도메인과 조회·집계 결과
- 방식: 동기 트랜잭션, 일부 비동기 소비 결과 저장
- 실패 처리: 행 잠금, 기존 결과 조회와 고유 식별자 기반 중복 방지
- 한계: Redis의 즉시 상태와 시간차가 생길 수 있음

## Batch

- 문제: 온라인 요청에서 반복 계산하기 무거운 지표와 위험 분석 분리
- 데이터: 입찰 관계 통계, 회원·계정쌍 위험, 서비스·카테고리 지표
- 방식: 예약 또는 관리자 수동 실행
- 실패 처리: Job 실행 상태 기록과 입력 검증, 페이지 반복 방어
- 한계: 위험 점수의 품질과 오탐은 별도 데이터 평가가 필요

## RabbitMQ와 Kafka를 나눈 이유

RabbitMQ 경로는 정해진 시점에 수행할 **작업 전달**이 목적입니다. Kafka 경로는 이미 발생한 입찰 결과를 남기는 **감사 이벤트 기록**이 목적입니다. 현재 프로젝트에서 두 기술은 같은 메시지를 중복 운반하지 않고 서로 다른 생명주기와 실패 정책을 담당합니다.
