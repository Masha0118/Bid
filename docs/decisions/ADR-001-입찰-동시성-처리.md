# Redis Lua로 입찰 판정과 상태 변경을 묶는다

## 상황

한 입찰은 최고가뿐 아니라 최고 입찰자, 입찰 기록, 참여자 수, 종료 시각과 버전을 함께 바꿉니다. 조회와 변경이 분리되면 동시 요청이 서로의 값을 덮을 수 있습니다.

## 검토한 선택지

- DB 트랜잭션: 영속 데이터와 묶기 쉽지만 별도 동시 수정 제어가 필요
- 비관적 락: 순서를 명확히 보장하지만 잠금 대기와 범위 관리가 필요
- 낙관적 락: 충돌이 적을 때 유용하지만 실패 재시도와 사용자 응답이 필요
- Redis Lua: Redis 런타임 상태의 여러 검증과 변경을 한 실행으로 처리

## 결정

입찰의 즉시 판정과 런타임 상태 변경을 Redis에서 실행되는 Lua로 처리합니다.

## 선택 이유

현재 경매의 즉시 상태가 Redis에 있고, 한 요청이 여러 Redis 자료구조를 함께 바꾸기 때문입니다. 특정 기술의 보편적 우위가 아니라 현재 데이터 위치와 변경 범위에 맞춘 선택입니다.

## 장점

- 검증과 상태 변경 사이에 다른 입찰이 끼어들지 않음
- 최고가, 입찰자, 버전, 기록과 자동 연장을 같은 결과로 반환
- 실패 이유를 안정적인 결과 코드로 구분

## 고려할 점

- PostgreSQL과는 하나의 트랜잭션이 아님
- Lua 변경 시 테스트와 결과 코드 호환성 관리 필요
- Redis 장애 복구와 장시간 스크립트 방지 필요

## 현재 검증 범위

실행 서비스와 판매자 입찰, 최소가, 시간 조건, 취소 후 재입찰, Redis 오류를 다루는 테스트 코드를 확인했습니다. 이번 분석에서는 테스트를 재실행하지 않았습니다.
