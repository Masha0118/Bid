# Kafka로 입찰 감사 이벤트를 기록한다

## 상황

성공·실패 입찰을 감사 목적으로 남겨야 하지만, 브로커나 DB 지연이 사용자 입찰 응답을 오래 막지 않아야 합니다.

## 검토한 선택지

- 입찰 요청 안에서 감사 DB에 동기 저장
- 애플리케이션 내부 비동기 큐만 사용
- Kafka 발행·소비와 로컬 실패 보관을 조합

## 결정

입찰 판정 후 감사 이벤트를 비동기 큐에 넣고 Kafka 브로커 ACK를 확인합니다. 소비자는 이벤트 식별자로 중복을 확인해 PostgreSQL에 저장합니다.

## 선택 이유

입찰 판정과 감사 기록의 처리 시간을 분리하면서, 발행 확인과 소비 중복 방지를 명시적으로 구현할 수 있기 때문입니다.

## 장점

- 사용자 응답 경로와 감사 저장 분리
- 발행 ACK 실패를 재시도 대상으로 구분
- 중복 소비를 기존 이벤트 확인과 DB 제약으로 흡수

## 고려할 점

- 내부 큐가 가득 차거나 재시도가 소진되면 로컬 실패 보관에 의존
- 로컬 파일은 여러 인스턴스가 공유하지 않음
- 보존 기간, 재처리 관찰과 개인정보 최소화 정책 필요

## 현재 검증 범위

ACK 실패, 재시도, 로컬 실패 보관·재생, 중복 소비와 실패 이벤트 분리의 테스트 코드 및 Testcontainers 통합 테스트를 확인했습니다. 이번 분석에서는 실행하지 않았습니다.
