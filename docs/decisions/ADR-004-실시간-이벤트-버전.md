# 버전 번호로 WebSocket 이벤트 순서를 확인한다

## 상황

WebSocket 메시지는 재연결과 네트워크 상황에 따라 중복되거나 늦게 도착하고, 일부가 누락될 수 있습니다. REST 응답도 더 최신인 WebSocket 상태보다 늦게 올 수 있습니다.

## 검토한 선택지

- 이벤트마다 전체 REST 재조회
- 도착 순서대로 무조건 반영
- 서버 버전과 클라이언트 마지막 버전을 비교하고 누락 시 전체 재조회

## 결정

Redis 상태 변경과 함께 버전을 증가시키고, 클라이언트가 마지막 버전과 이벤트 구간을 비교합니다.

## 선택 이유

정상 이벤트에서는 바뀐 값만 반영하면서도, 중복·역순·누락을 클라이언트가 판정할 수 있기 때문입니다.

## 장점

- 이미 처리한 이벤트를 무시
- 정상 구간은 전체 조회 없이 반영
- 누락이 확인되면 기준 상태를 다시 조회
- 늦은 REST 응답이 최신 화면을 덮는 상황 방지

## 고려할 점

- 버전이 모든 서버 인스턴스에서 동일한 기준으로 증가해야 함
- 전체 재조회가 반복되지 않도록 동시 실행 제어 필요
- 실제 네트워크 환경의 재연결 시험 필요

## 현재 검증 범위

서버 버전 생성, 버퍼 구간 전달, Flutter의 중복·누락·늦은 REST 판정과 단위 테스트 코드를 확인했습니다. 실제 네트워크 장애 통합 시험 결과는 확인하지 못했습니다.
