# RabbitMQ로 경매 시작·종료 작업을 전달한다

## 상황

상품 등록 요청과 실제 경매 시작·종료 시점은 다릅니다. 애플리케이션 메모리의 타이머만 사용하면 서버 재시작과 여러 인스턴스에서 작업 소유권이 불명확해집니다.

## 검토한 선택지

- 프로세스 내부 스케줄러만 사용
- DB를 주기적으로 조회
- 지연·예약 메시지를 사용하고 누락을 주기적 재탐색으로 보완

## 결정

Main API가 시작·종료 작업 메시지를 발행하고 Auction Server가 소비합니다. 종료 누락은 별도 주기적 만료 경매 탐색으로 보완합니다.

## 선택 이유

사용자 요청과 미래 시점의 작업 실행을 분리하고, 소비 실패를 메시지 재시도 정책으로 다룰 수 있기 때문입니다.

## 장점

- 상품 등록 응답과 미래 작업 실행을 분리
- 경매 서버가 시작·종료 책임을 집중 관리
- 소비 예외를 숨기지 않고 재시도 경로로 전달

## 고려할 점

- 예약 메시지 누락과 브로커 장애를 완전히 배제할 수 없음
- 주기적 재탐색과 DB·Redis 복원 경로를 함께 검증해야 함
- 실제 메시지 이름과 지연 구현은 인프라 세부사항이므로 공개하지 않음

## 현재 검증 범위

발행, 시작·종료 리스너, 재시도·실패 보관 구성과 관련 테스트 코드를 확인했습니다. 브로커 장애 통합 시험은 이번 분석에서 확인하지 못했습니다.
