# 현재 범위와 개선

## 현재 웹 체험 범위

- 새 경매 준비와 테스트 데이터 정리
- 금액 입력을 통한 직접 입찰
- 2~10명의 가상 참가자 동시 입찰
- Redis Lua 기반 입찰 판정
- 네 참가자의 WebSocket 상태 동기화
- 실제 요청 변화에 따른 처리 과정 시각화
- 평균·p95 응답 시간과 처리량 확인
- 완료 요청 수와 참가자 Version 일치 확인
- 고급 성능 설정과 최종 상태 비교

## 현재 확인 범위

- Kafka는 전달 요청 단계까지 화면에서 확인합니다.
- WebSocket은 네 참가자의 현재 연결과 Version을 확인합니다.
- RabbitMQ 예약 종료와 거래 생성은 기본 체험에 포함되지 않습니다.
- 성능 수치는 해당 테스트 환경의 전체 실행 결과로 해석합니다.
- 현재 배포는 화면 접속만 확인됐고 실행 기능 연결은 추가 확인이 필요합니다.

## 개선 방향

### 더 긴 실행과 더 많은 참가자

장시간 테스트, 더 많은 동시 참가자와 다중 경매 시나리오를 분리해 메모리, 연결 안정성과 처리량 변화를 확인합니다.

### 부하 발생기 분리

부하 발생기와 서버를 다른 장비 또는 실행 단위로 분리하고 각 사양을 기록해 서버 처리와 발생기 한계를 구분합니다.

### 다중 서버 환경

여러 Auction Server에서 Redis 상태, WebSocket 전달 순서와 Version 수렴이 일관된지 검증합니다.

### 장애 주입

Redis, Kafka, RabbitMQ와 PostgreSQL 연결 실패를 주입해 재시도, 중복 방지와 최종 상태 수렴을 확인합니다.

### WebSocket 연결 복구

연결 강제 종료, 이벤트 지연·중복·누락을 재현하고 최신 상태 재조회 횟수와 복구 시간을 지표로 남깁니다.

### 메시징 완료 상태

Kafka 전달 요청 이후 소비·저장 완료와 오류를 실행별로 연결해 `확인 정보 없음` 범위를 줄입니다.

### 경매 종료 후 전체 흐름

예약 종료, 낙찰 결과, 거래 생성, 중복 요청과 재처리를 하나의 체험·검증 시나리오로 연결합니다.
