# Bid On 프로젝트 핵심 요약

실시간 입찰부터 낙찰 이후 거래까지 연결한 경매 서비스의 설계·구현·검증 범위를 약 5분 안에 확인할 수 있도록 정리한 문서입니다. 운영 소스코드와 실제 앱 화면은 공개하지 않습니다.

## 1. 프로젝트 개요

Bid On은 상품 등록, 경매 시작 예약, 실시간 입찰, 마감 임박 자동 연장, 낙찰 또는 유찰, 후속 거래·채팅·알림을 연결한 경매 서비스입니다. 여러 사용자가 같은 상태를 바꾸는 순간의 정확성과 비동기 후속 처리 실패를 핵심 문제로 다뤘습니다.

```text
상품 등록 → 경매 예약 → 실시간 입찰 → 종료 판정 → 낙찰 거래 → 후속 커뮤니케이션
```

## 2. 담당 범위

비공개 원본 저장소를 기준으로 Flutter 사용자 앱, Main API, 실시간 경매 서버, 관리자 서버·웹과 Batch 서버를 개발했습니다. 이 공개 저장소에는 운영 소스 대신 프로젝트 구조, 기술적 의사결정, 검증 범위와 공개용 시각화 도구를 정리했습니다.

개인별 기여율이나 구현량을 공개 문서만으로 증명할 수 없어 임의의 수치는 기재하지 않습니다.

## 3. 시스템 구성

| 구성요소 | 책임 |
|---|---|
| Flutter App | 경매 조회·입찰·거래 사용자 흐름 |
| Main API | 회원·상품·거래와 영속 상태 |
| Auction Server | 입찰 판정, 실시간 전달, 종료 처리 |
| Redis + Lua | 런타임 상태와 입찰의 원자적 검증·변경 |
| RabbitMQ | 미래 시점의 경매 시작·종료 작업 전달 |
| Kafka | 입찰 성공·실패 감사 이벤트 기록 |
| PostgreSQL | 상품·거래·종료 결과·감사 데이터 영속화 |
| Admin · Batch | 관리 기능과 집계·위험 분석 |

Redis Lua는 입찰 판정과 런타임 상태 변경까지만 묶습니다. WebSocket, Kafka, Main API 동기화는 그 이후의 후속 처리이며 PostgreSQL과 하나의 트랜잭션을 구성하지 않습니다.

## 4. 해결한 핵심 문제

### 동시 입찰

경매 상태 확인, 판매자·현재 최고 입찰자·최소가 조건, 최고가·최고 입찰자·입찰 횟수·자동 연장·Version 변경을 Redis Lua 안에서 처리했습니다. 두 요청이 같은 이전 값을 읽고 덮어쓰는 구조를 피하고, Redis가 요청을 순서대로 판정하게 했습니다.

### 실시간 중복과 누락

최초 진입은 전체 조회, 이후는 WebSocket 변경 정보 반영으로 나눴습니다. 수신 Version이 이미 처리한 값이면 무시하고, 바로 다음 값이면 반영하며, 중간 값이 비면 전체 상태를 다시 조회합니다. 오래된 전체 조회 응답이 최신 화면을 덮지 않도록 Version을 다시 비교합니다.

### 종료 중복과 후속 실패

경매별 잠금, 기존 완료 결과, 최종 낙찰 정보를 차례로 확인합니다. Main API도 기존 거래를 확인해 같은 결과를 기존 거래로 수렴시킵니다. 후속 반영 실패 시 런타임 상태를 성공 전에 정리하지 않아 재처리 근거를 유지하지만, 모든 장애에서 자동 복구된다고 주장하지 않습니다.

## 5. 주요 기술 선택

- Redis Lua: 입찰 판정과 관련 상태 변경의 원자적 범위 확보
- WebSocket + Version: 정상 상황은 변경 정보만 반영하고 누락 때만 전체 재조회
- RabbitMQ: 예약된 시작·종료 작업 전달
- Kafka: 사용자 응답과 분리한 입찰 감사 기록
- PostgreSQL 잠금·기존 결과 조회: 낙찰 거래 중복 방지

기술을 사용했다는 사실보다 각 기술의 책임과 실패 경계를 구분하는 데 초점을 뒀습니다.

## 6. 구현 및 검증 범위

원본 프로젝트의 입찰 규칙, 자동 연장, WebSocket Version 판정, 종료·거래 중복 방지와 메시징 경로는 `코드에서 구현 확인` 상태입니다. 관련 자동화 테스트 파일은 `테스트 코드 확인` 상태이며 이번 공개 저장소 작업에서는 실행하지 않았습니다.

공개 Flow Console은 `흐름 재구성` 상태이고 type-check, 딥링크 테스트, 빌드를 이번 작업에서 직접 실행해 `테스트 실행 확인`으로 구분합니다. 실제 네트워크, 장애, 동시 부하와 다중 인스턴스 통합 시험은 `추가 검증 필요`입니다.

## 7. Flow Console

[Bid On Flow Console](https://masha0118.github.io/Bid/)은 정상 입찰, 동시 입찰, 자동 연장, 이벤트 누락, 중복 종료, 거래 후속 처리 실패를 가상 데이터로 재생합니다. 실제 Flutter UI나 운영 모니터링 화면이 아니며 실제 서버에 요청하지 않습니다.

각 시나리오는 기술 문서로, 주요 기술 문서는 해당 시나리오 딥링크로 연결됩니다. [Flow Console 설계와 공개 기준](12-flow-console.md)에서 데이터 기준과 한계를 확인할 수 있습니다.

## 8. 현재 한계

- Redis와 PostgreSQL 사이의 시간차와 장기 실패 재처리 통합 시험 필요
- Redis·브로커 재시작과 네트워크 단절을 묶은 장애 시험 필요
- 실제 WebSocket 장시간 재연결·다중 서버 전환 검증 필요
- Kafka 실패 이벤트의 다중 인스턴스 공유 저장 방식 검토 필요
- 성능 수치와 동시 사용자 수를 제시할 재현 가능한 결과 없음
