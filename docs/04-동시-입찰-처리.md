# 동시 입찰 처리

[입찰 처리 다이어그램](../diagrams/bid-processing.mmd) · [웹에서 직접 확인](https://bid-on-performance-lab.pages.dev/)

## 1. 문제 상황

두 참가자가 같은 현재가를 보고 거의 동시에 입찰하면 두 요청 모두 자신이 유효하다고 판단할 수 있습니다. 애플리케이션이 값을 읽고 조건을 확인한 뒤 다시 저장하는 동안 다른 요청이 끼어들면 최고가, 최고 입찰자와 입찰 횟수가 서로 맞지 않을 수 있습니다.

## 2. 단순 조회 후 갱신의 위험

```text
요청 A: 현재가 조회 ─ 조건 통과 ─ 저장
요청 B: 현재가 조회 ─ 조건 통과 ─ 저장
```

두 요청이 같은 이전 값을 읽으면 나중 저장이 앞선 결과를 덮을 수 있습니다. 데이터베이스 잠금만으로 해결할 경우 실시간 런타임 상태를 Redis에서 관리하는 구조와 판정 기준이 나뉠 수 있습니다.

## 3. Redis Lua를 선택한 이유

Redis는 Lua Script 한 번의 실행 동안 다른 명령이 중간에 끼어들지 않도록 처리합니다. Bid On은 입찰 조건 확인과 관련 상태 변경을 같은 실행에 배치해 읽기와 쓰기 사이의 경쟁 구간을 줄였습니다.

## 4. 한 실행 안에서 판정하는 상태

- 경매 진행 여부와 종료 시각
- 판매자 본인 입찰 여부
- 현재 최고 입찰자 여부
- 최소 다음 입찰가 충족 여부
- 최고가와 최고 입찰자
- 입찰 횟수와 Version
- 자동 연장 조건과 종료 시각

전체 Lua 코드를 문서에 복사하지 않고 판정 범위와 상태 변화만 설명합니다.

## 5. 입찰 반영 흐름

1. 요청과 경매 식별 정보를 검증합니다.
2. Redis의 현재 경매 상태를 읽습니다.
3. 입찰 조건을 확인합니다.
4. 최고가, 최고 입찰자, 입찰 횟수와 Version을 변경합니다.
5. 필요하면 종료 시각을 연장합니다.
6. 반영 결과를 Auction Server에 반환합니다.
7. WebSocket과 Kafka 후속 처리를 시작합니다.

## 6. 조건상 미반영 흐름

다음은 시스템 장애가 아니라 경매 규칙에 따른 정상 결과입니다.

- 현재 입찰가보다 낮음
- 이미 최고 입찰자인 참가자의 요청
- 시작 전 또는 종료 후 요청
- 판매자 본인의 요청

조건상 미반영 요청은 현재가와 Version을 바꾸지 않습니다. 화면에서는 사유별 건수를 분리해 확인할 수 있습니다.

## 7. 오류 흐름

Redis 연결 실패, 제한 시간 초과나 예상하지 못한 서버 오류는 `시스템 오류` 또는 `응답 시간 초과`로 구분합니다. 정상 경매 조건과 시스템 오류를 합쳐 성공률처럼 해석하지 않습니다.

## 8. Version 변경

Version은 상태가 유효하게 변경된 순서를 나타냅니다. 반영된 입찰은 Version을 증가시키고, 조건상 미반영 결과는 증가시키지 않습니다. WebSocket 수신 화면은 Version을 기준으로 최신 상태 여부를 판단합니다.

## 9. 웹에서 확인하는 방법

1. 새 테스트를 시작합니다.
2. 가상 참가자를 경쟁 패턴과 빠른 속도로 설정합니다.
3. 직접 입찰을 함께 실행합니다.
4. 반영된 입찰과 조건상 미반영 건수를 비교합니다.
5. 현재가, 최고 입찰자, 입찰 횟수와 Version을 확인합니다.

## 10. 현재 한계

- Lua의 원자성은 Redis 내부 판정과 상태 변경에만 적용됩니다.
- WebSocket, Kafka와 PostgreSQL은 별도 후속 처리입니다.
- 네트워크 단절·Redis 재시작·다중 서버 전환을 묶은 장애 검증은 추가 과제입니다.
- 웹 화면의 짧은 실행 결과는 장시간·대규모 처리 한계를 의미하지 않습니다.
