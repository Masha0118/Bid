# 동시 입찰 처리

흐름도 원본은 [입찰 처리 흐름](../diagrams/입찰-처리-흐름.mmd)에 있습니다.

## 문제 상황

현재 최고가가 10,000원일 때 A가 11,000원, B가 12,000원을 거의 동시에 입찰한다고 가정합니다. 두 요청이 각각 10,000원을 읽은 뒤 따로 저장하면 마지막 저장이 앞선 결정을 덮어쓸 수 있습니다.

이때 서로 어긋날 수 있는 값은 현재 최고가, 최고 입찰자, 입찰 횟수, 종료 시각, 입찰 기록입니다. 특히 마감 직전이라면 가격만 맞고 자동 연장 여부나 버전이 틀리는 상태도 가능합니다.

## 검토 가능한 선택지

| 방식 | 장점 | 고려할 점 |
|---|---|---|
| 단순 DB 트랜잭션 | 영속 데이터와 한 트랜잭션으로 묶기 쉬움 | 동시 수정 자체를 직렬화하지 않으면 갱신 충돌이 남음 |
| 비관적 락 | 같은 행의 변경 순서를 명확히 제어 | 잠금 대기와 범위 관리가 필요 |
| 낙관적 락 | 충돌이 적을 때 잠금 대기를 줄일 수 있음 | 충돌 시 재시도 정책과 사용자 응답 설계가 필요 |
| Redis Lua | Redis 안의 여러 검증·변경을 한 번에 실행 | Redis와 DB의 책임 분리, 스크립트 유지보수와 복구 설계 필요 |

실제 벤치마크 결과가 없으므로 특정 방식이 항상 더 빠르다고 주장하지 않습니다.

## 선택한 방식

실시간 경매 상태가 Redis에 있고, 한 입찰이 여러 값을 함께 바꿔야 하는 현재 구조에 맞춰 Redis에서 실행되는 Lua를 선택했습니다. 스크립트 안에서 다음을 확인하거나 변경합니다.

- 경매 상태 존재와 시작·종료 시각
- 판매자 본인, 취소 후 재입찰, 현재 최고 입찰자의 연속 입찰 조건
- 현재 최고가와 입찰 단위로 계산한 최소 입찰가
- 최고가와 최고 입찰자
- 마감 임박 자동 연장 시 종료 시각과 연장 횟수
- 경매 버전, 입찰 기록, 입찰 횟수와 참여자 집합
- 성공 또는 실패 결과 코드

```text
검증
→ Redis 상태 변경
→ 입찰 기록과 버전 저장
→ 결과 반환
→ Main API 동기화·WebSocket·알림·Kafka 후속 처리
```

마지막 단계는 Lua의 원자 처리 범위가 아닙니다. 코드에서는 현재가 동기화, 입찰 영속 기록, 이전 최고 입찰자 알림과 감사 이벤트 발행을 별도 비동기 경로로 넘깁니다.

## 구현 범위와 검증 방법

- 구현: Lua 원문을 Java 리소스로 실행하고 반환 코드를 서비스 결과로 변환
- 테스트 코드: 판매자 입찰, 시작 전·종료 후 입찰, 최소가 미충족, 연속 입찰, 취소 후 재입찰, Redis 실패 코드, 성공 후 후속 처리 확인
- 이번 분석: 테스트 파일과 구현은 확인했지만 원본을 읽기 전용으로 유지하기 위해 테스트 실행은 하지 않음

## 한계

- Redis와 PostgreSQL은 하나의 트랜잭션이 아니므로 후속 동기화 실패를 관찰하고 재처리해야 합니다.
- Redis 장애 시 입찰은 안정적인 실패 코드로 거절하지만, 복구 이후 상태 정합성은 별도 확인이 필요합니다.
- Lua가 커질수록 변경 영향과 결과 코드 관리가 어려워질 수 있습니다.
- 장시간 실행되는 Lua는 Redis의 다른 명령을 막을 수 있어 연산 범위를 짧게 유지해야 합니다.
- 거래와 감사 기록의 최종 영속성은 PostgreSQL 경로가 담당합니다.
