# 프로젝트 개요

## 서비스 정의

실시간 경매 시스템은 상품 등록부터 실시간 입찰, 낙찰, 거래·채팅·알림, 관리자 검토까지 연결하는 경매 서비스입니다. 사용자 앱과 관리 화면을 분리하고, 실시간 입찰의 일관성은 별도 경매 서버가 담당합니다.

## 기획 배경

경매의 핵심은 단순한 가격 수정이 아니라 여러 사용자의 동시 요청을 하나의 순서로 판정하고, 화면·메시지·영속 데이터가 그 결과를 따라가게 만드는 것입니다. 프로젝트는 이 경계에서 발생하는 동시성, 중복 처리, 실패 복구와 관리 기능을 직접 구현하는 데 초점을 맞췄습니다.

## 사용자 역할

- 일반 사용자: 상품 등록, 입찰, 낙찰 이후 거래, 채팅, 알림, 신고
- 판매자: 경매 조건 설정, 낙찰 이후 거래 진행
- 관리자: 광고·신고·분쟁·회원 제재 검토, 위험 분석·지표 확인

## 핵심 사용자 흐름

상품 등록 → 예약된 경매 시작 → 실시간 입찰 → 자동 연장 가능성 판정 → 종료 → 낙찰 또는 유찰 → 거래·채팅·알림 → 관리자 사후 검토

## 주요 기능

- Redis Lua 기반 입찰 규칙 확인과 원자적 상태 변경
- WebSocket으로 변경된 경매 상태 전달
- 버전 비교를 통한 중복·역순·누락 이벤트 처리
- 종료 잠금, 완료 결과 확인과 낙찰 거래 중복 방지
- RabbitMQ 예약 작업과 Kafka 감사 이벤트의 책임 분리
- JWT 인증, 내부 서버 요청 인증, 요청 제한과 데이터 소유권 확인
- 관리자 광고 심사·신고·분쟁·회원 제재·모니터링
- Batch 지표 집계와 판매자·입찰자 관계 기반 위험 분석

## 직접 담당한 범위

Git 이력은 Main API, 경매 서버, Flutter 앱, 관리자 서버·웹, Batch 서버에 걸친 지속적인 변경을 보여줍니다. 다만 공동 작업자별 기여를 객관적으로 분리할 별도 자료는 확인하지 못해, 공개 문서에서는 개인 기여율을 주장하지 않습니다.

## 서버와 클라이언트 구성

| 구성 | 책임 |
|---|---|
| Flutter App | 사용자 경매·거래·채팅 화면과 실시간 상태 반영 |
| Vue 사용자 웹 | 초기 사용자 웹 클라이언트 |
| Main API | 사용자, 상품, 거래, 채팅, 알림과 영속 데이터 |
| Auction Server | 실시간 입찰 판정, WebSocket, 종료와 복구 |
| Admin Server + Vue Admin | 심사, 제재, 분쟁, 위험 분석 조회와 모니터링 |
| Batch Server | 일별 지표와 입찰 관계 기반 위험 분석 |

## 핵심 기술 스택과 확인된 버전

| 영역 | 확인된 정보 |
|---|---|
| Backend | Java 17, Spring Boot 3.4.1 |
| Persistence | Spring Data JPA/JDBC, PostgreSQL 드라이버, Flyway |
| Realtime | Spring WebSocket, Redis, Redis Lua |
| Messaging | Spring AMQP, Spring Kafka |
| Batch | Spring Batch 3.4.1 의존성 관리 범위 |
| User App | Flutter, Dart SDK 제약 `^3.8.1` |
| Admin Web | Vue 3.5.13, Pinia 2.3.1, Vite 5.4.19 |
| User Web | Vue 3.5.13, Pinia 2.3.0, Vite 8.0.16 |

버전은 빌드 정의에서 직접 확인된 값만 적었습니다. PostgreSQL 서버 버전처럼 실행 환경에 의존하는 값은 공개 기술 스택에서 확정하지 않았습니다.

## 프로젝트 범위에서 제외한 내용

- 실제 인프라 주소와 배포 구성
- 결제 사업자·소셜 로그인·푸시 설정의 실제 값
- 내부 보안 임계값과 위험 점수 계산식
- 미완성 여부를 별도로 판단해야 하는 AI 고객센터·AI 상품 등록의 품질 주장
- 운영 환경의 가용성·성능 수치
