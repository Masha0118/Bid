# 구현과 검증 상태

| 기능 | 상태 | 근거와 범위 |
|---|---|---|
| 직접 입찰 | 직접 체험 기능 구현 · 배포 연결 확인 필요 | 웹 입력과 서버 입찰 중계 구현, 현재 배포는 실행 비활성 상태 |
| 가상 참가자 | 직접 체험 기능 구현 · 배포 연결 확인 필요 | 2~10명, 세 속도와 세 패턴 구현 |
| Redis Lua 판정 | 코드와 자동화 테스트 확인 | 조건 확인과 상태 변경의 Redis 내부 원자 범위 |
| WebSocket 참가자 동기화 | 격리 환경에서 실행 확인 | 네 참가자 구독과 현재가·최고 입찰자·Version 동기화 |
| Version 처리 | 코드와 문서에서 확인 | 중복·역순·누락 판정, 강제 재연결 통합 검증은 추가 필요 |
| 처리 과정 시각화 | 격리 환경에서 실행 확인 | 실제 요청 수 변화와 수집된 처리 단계 사용 |
| 응답 시간 측정 | 직접 확인 기능 구현 | 평균·p95, 고급 화면의 추가 백분위 지표 |
| 처리량 측정 | 직접 확인 기능 구현 | 현재 및 평균 초당 요청 수, 환경 전체 결과로 해석 |
| 최종 상태 확인 | 부분 확인 | 최고가·최고 입찰자·입찰 횟수·Version 비교 |
| Kafka 이벤트 전달 | 부분 확인 | 전달 요청 단계 관측, 실행별 최종 저장 완료 수는 확인 정보 없음 |
| Kafka 중복 소비 방지 | 자동화 테스트 확인 | 소비·중복 처리 테스트 존재 |
| RabbitMQ 예약 작업 | 코드와 자동화 테스트 확인 | 시작·종료 작업 전달, 기본 체험 미포함 |
| 중복 종료 방지 | 자동화 테스트 확인 | 잠금, 기존 결과와 거래 중복 방지 테스트 존재 |
| 종료 실패 재처리 | 부분 확인 | 성공 전 상태 유지 로직 확인, 전체 장애 통합 검증 필요 |
| 테스트 데이터 정리 | 격리 환경에서 실행 확인 | 세션 종료와 연속 실행 정리 검증 기록 확인 |
| 장시간·대규모 부하 | 추가 검증 필요 | 재현 가능한 최신 결과 파일 없음 |
| 다중 서버 장애 복구 | 현재 범위 미포함 | 장애 주입과 상태 수렴 통합 검증 필요 |

## 상태 해석

- 직접 체험 가능: 현재 배포에서 사용자 동작까지 확인된 경우
- 격리 환경에서 실행 확인: 제한된 통합 환경의 실행 기록을 확인한 경우
- 자동화 테스트 확인: 관련 테스트 코드가 존재하는 경우
- 코드와 문서에서 확인: 구현 구조는 확인했지만 이번 작업에서 실행하지 않은 경우
- 부분 확인: 경로 일부만 관측하거나 필요한 결과가 모두 수집되지 않는 경우
- 추가 검증 필요: 구현 여부와 별개로 통합·장시간 검증이 필요한 경우
- 기본 체험 미포함: 상세 흐름에는 존재하지만 기본 화면에서 실행하지 않는 경우
- 현재 범위 미포함: 다음 단계의 설계·검증 과제인 경우
