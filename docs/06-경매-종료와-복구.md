# 경매 종료와 복구

흐름도 원본은 [경매 종료 복구 흐름](../diagrams/경매-종료-복구-흐름.mmd)에 있습니다.

## 정상 흐름

1. RabbitMQ 종료 메시지 또는 주기적 재탐색이 종료 시간이 지난 경매를 찾습니다.
2. 경매별 Redis 잠금으로 동시에 들어온 종료 작업을 한 번에 하나만 진행합니다.
3. PostgreSQL에 이미 종료 결과 요약이 있는지 확인합니다.
4. Redis 입찰 기록을 우선 사용하고, 없으면 영속 입찰 기록을 조회해 최종 입찰자와 금액을 결정합니다.
5. Auction Server가 Main API에 낙찰 또는 유찰 결과를 전달합니다.
6. Main API는 상품 행을 잠그고 기존 거래를 조회한 뒤 거래를 생성하거나 기존 거래를 반환합니다.
7. 결과 요약과 입찰 기록을 저장하고 완료 상태를 전파한 뒤 런타임 상태를 정리합니다.

핵심 원칙은 **낙찰 결과와 거래 생성이 성공한 뒤에만 복구에 필요한 런타임 상태를 삭제하는 것**입니다.

## 종료 요청이 중복된 경우

같은 경매에 여러 작업이 실행되면 첫 작업만 잠금을 얻습니다. 이미 완료 결과가 있으면 Main API를 다시 갱신하지 않고 남은 런타임 상태만 정리합니다. Main API도 기존 낙찰 거래를 먼저 조회하며, 같은 결과는 기존 거래로 수렴시키고 다른 낙찰자·금액은 거절합니다.

관련 테스트 코드에는 종료 재시도, 잠금 보유 중 건너뛰기, Main API 중복 갱신 방지와 거래 1회 생성 시나리오가 있습니다.

## 후속 거래 생성이 실패한 경우

Auction Server가 낙찰 결과를 결정했더라도 Main API가 성공으로 응답하지 않으면 종료 결과 요약 저장과 Redis 런타임 상태 삭제를 진행하지 않습니다. 주기적 재탐색이나 재전달로 다시 처리할 수 있는 상태를 남깁니다.

알림은 낙찰 결과 반영 후 별도 호출되므로 알림 실패가 거래 생성 자체를 되돌리지는 않습니다. 알림의 완전한 재처리 보장은 코드에서 확인하지 못했습니다.

## 종료 처리가 누락된 경우

- 예약 메시지가 누락되거나 서버가 재시작된 경우, 주기적으로 Redis의 만료 경매를 다시 탐색합니다.
- 서버 시작 시 영속 백업을 기준으로 누락된 Redis 경매 상태를 복원하는 실행기가 있습니다.
- 복원 시 오래된 백업에 일부 필드가 없어도 기본 입찰 단위와 영속 버전을 유지하는 테스트 코드가 있습니다.

## 현재 한계

- Redis 잠금은 TTL 기반이므로 처리 시간이 TTL을 넘는 상황과 네트워크 분할을 추가 검증해야 합니다.
- 알림·아카이브 같은 일부 후속 작업은 낙찰 거래와 하나의 트랜잭션이 아닙니다.
- DB 백업과 Redis 상태 복원은 구현됐지만 Redis 재시작을 포함한 전체 복구 시험 결과는 이번 분석에서 확인하지 못했습니다.
