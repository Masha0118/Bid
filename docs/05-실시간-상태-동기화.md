# 변경된 경매 정보만 실시간 반영

> [Flow Console에서 이벤트 누락과 상태 복구 보기](https://masha0118.github.io/Bid/?scenario=websocket-recovery)
>
> 이 화면은 공개용 가상 데이터로 재구성한 기술 시연이며, 실제 구현 및 검증 범위는 현재 문서를 기준으로 합니다.

흐름도 원본은 [실시간 버전 흐름](../diagrams/실시간-버전-흐름.mmd)에 있습니다.

## 기존 방식

WebSocket 이벤트 수신 → 상품 전체 REST 재조회 → 전체 화면 상태 갱신

이 방식은 이해하기 쉽지만 입찰이 올 때마다 동일한 데이터를 다시 조회하고, 늦게 도착한 REST 응답이 더 최신 WebSocket 상태를 덮을 수 있습니다.

## 개선 방식

최초 진입 시 전체 상태 조회 → 이후 바뀐 값만 WebSocket으로 전달 → 버전 번호 비교 → 정상 이벤트 반영 → 중복·늦은 이벤트 무시 → 중간 이벤트 누락 시 REST 전체 조회

서버는 입찰 상태 변경과 같은 원자 범위에서 경매 버전을 증가시킵니다. 클라이언트는 상품별 마지막 버전과 이미 본 입찰 키를 관리합니다.

## 버전 번호로 이벤트 순서 확인

현재 화면 Version 10

- Version 9 → 10: 이미 반영된 이벤트이므로 무시
- Version 10 → 11: 정상적으로 이어지므로 바뀐 값만 반영
- Version 11 → 12만 도착: 중간 이벤트가 누락됐으므로 전체 상태 재조회

구현에서는 여러 입찰을 한 번에 전달하는 버퍼 구간도 시작 버전과 끝 버전으로 판정합니다. 늦게 온 REST 결과의 버전이 현재 화면보다 낮으면 화면을 덮지 않습니다.

## 확인한 코드 흐름

1. Flutter 화면 진입 시 REST로 경매 스냅샷과 입찰 목록을 채웁니다.
2. WebSocket 메시지를 파싱해 상품과 버전 범위를 확인합니다.
3. 이미 처리한 범위면 무시합니다.
4. 현재 버전 다음으로 연결되면 새 입찰만 합칩니다.
5. 버전 간격이 있거나 서버가 재동기화를 요구하면 한 번만 전체 재조회를 예약합니다.
6. REST 응답과 그 사이 도착한 WebSocket 입찰을 버전 기준으로 합칩니다.

## 검증 범위와 한계

단위 테스트에는 동일 메시지 중복 수신, 버전 누락, 연속 구간, WebSocket보다 늦은 REST 응답, 초기 조회 중 도착한 입찰을 포함합니다. 서버 측에는 다중 인스턴스 Pub/Sub와 버퍼 순서를 확인하는 테스트 코드가 있습니다.

이번 작업에서는 테스트를 재실행하지 않았고, 실제 네트워크 단절·재연결을 반복하는 통합 시험 결과도 확인하지 못했습니다. 따라서 코드와 테스트 시나리오는 구현됐지만 실제 네트워크 환경 검증은 추가로 필요합니다.
