# 실시간 상태 동기화

[상태 동기화 다이어그램](../diagrams/realtime-sync.mmd) · [웹에서 직접 확인](https://bid-on-performance-lab.pages.dev/)

## 동기화 흐름

1. 화면 진입 시 경매의 전체 상태를 조회합니다.
2. 경매별 WebSocket 채널을 구독합니다.
3. 입찰이 반영되면 현재가, 최고 입찰자, 입찰 횟수와 Version 변경 이벤트를 받습니다.
4. 수신 Version과 화면의 현재 Version을 비교합니다.
5. 정상적인 다음 Version이면 변경을 반영합니다.
6. 중복·역순 이벤트는 최신 화면을 덮지 않도록 무시합니다.
7. Version이 건너뛰면 최신 전체 상태 재조회가 필요합니다.

## Version 판정

| 수신 상태 | 의미 | 처리 방향 |
|---|---|---|
| 수신 Version ≤ 현재 Version | 중복 또는 늦게 도착한 이벤트 | 무시 |
| 수신 Version = 현재 Version + 1 | 정상적인 다음 변경 | 반영 |
| 수신 Version > 현재 Version + 1 | 중간 이벤트 누락 가능성 | 최신 상태 재조회 |

전체 조회 응답이 늦게 도착할 수 있으므로 응답을 적용하기 직전에도 Version을 비교해야 합니다. 오래된 조회 결과가 더 최신인 WebSocket 상태를 덮지 않게 하는 것이 목적입니다.

## 참가자 카드

기본 체험의 네 참가자 카드는 경매 서버의 WebSocket을 각각 구독하는 상태 확인 화면입니다. 각 카드에는 다음 값이 표시됩니다.

- 연결 여부
- 현재가
- 최고 입찰자
- Version
- 마지막 갱신 시각

모든 카드가 연결되고 같은 Version을 표시하면 동일한 최신 상태를 보고 있는지 확인할 수 있습니다. 브라우저에는 서버 측 구독 결과가 전달됩니다.

## 연결이 끊겼을 때

연결 전에는 `연결 확인 중` 또는 `연결 중`으로 표시합니다. Version 차이를 이용한 누락 감지와 최신 상태 재조회 설계가 존재하지만, 기본 체험에서 WebSocket을 강제로 끊고 자동 복구하는 시나리오는 완료 기능으로 표시하지 않습니다.

## 현재 한계

- 참가자 카드의 연결 상태는 네 개의 제한된 구독을 대상으로 합니다.
- 재연결 횟수, 병합·대기·오류 수는 실행별 확인 정보가 없습니다.
- 실제 네트워크 지연과 다중 Auction Server 전환 검증은 추가 과제입니다.
- 장시간 연결 안정성은 짧은 체험 결과로 판단하지 않습니다.
